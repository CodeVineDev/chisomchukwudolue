Document Overview
       The Functional Requirements Document (FRD) describes the functional specifications required to improve Temu’s ordering system, with the goal of reducing customer complaints, minimizing incorrect size deliveries, and improving operational efficiency. This document outlines the specific system features and behaviours needed to enhance customer satisfaction and support accurate order processing.
 
Purpose and Scope
Purpose
       The Functional Requirements Document (FRD) was created to describe the functional specifications required to improve Temu’s ordering system and reduce customer complaints. It serves as a reference document to guide the development process, define the steps required for the system improvement and monitor progress throughout the project. This document is intended for use by the Project manager, Product manager, App Designer and Engineering team.


Scope
In Scope
1.	Redesign of the app user interface to improve navigation and usability.
2.	Enhancement of the payment and order system to ensure accurate order processing and secure transaction completion.
3.	Improvement of the size selection feature to enable customers choose correct product sizes and reduce incorrect order complaints.

Out of Scope
1.	Enhancement of product showcase are outside the scope of this project.
2.	Changing of the company logo is outside the scope of this project.
3.	Changes to the app colour scheme and visual branding are outside the scope of this project.		 
Functional Requirement
1.	Product Selection Accuracy Feature
•	The system shall display product sizes, colours and specifications clearly and visibly on every product page.
•	The system shall provide a product guide that enables customer to select the correct size and specifications before placing an order.
•	The system shall prevent customers from completing a purchase without fully selecting all required product options such as size, colour and specifications.
2.	Order Verification
•	The system shall display a full order summary including product details, size, colour quantity, and total amount for customer review before confirming purchase
•	The system shall automatically calculate applicable taxes and fees and display the total amount clearly before checkout completion. A confirmation email shall be sent to the customer’s registered email address upon successful order placement.
•	The system shall decline payment if an incorrect pin or authentication detail is entered, and shall temporarily lock the account after three consecutive failed attempts for security purposes.
3.	Payment Processing
•	The system shall support only card payment and direct bank transfers as accepted payment methods, ensuring secured and verified transactions.
•	The system shall allow maximum of three payment attempts before temporarily suspending the account pending identity verification and reactivation.
•	The system shall process refunds to the customer’s original payment method upon confirmation that the returned product is unused, undamaged and returned within the 7-day return window in line with Temu’s return policy.
 
4.	System Testing & Validation
•	The system shall undergo comprehensive testing across all features including size selection, order verification, payment processing and refund management before deployment.
•	The system shall support user acceptance testing (UAT) to ensure all features meet business and functional requirement before go-live.
•	The system shall be retested after any bug fix or modification to confirm that existing features remains functional and no new issues have been introduced.
5.	Order Tracking
•	The system shall display real-time order status including current location, shipping confirmation and estimated delivery date for every order placed.
•	The system shall automatically update the order status at each stage of the delivery process from order confirmation through to dispatch and final delivery.
•	 In the event of a delivery delay, the system shall notify the customer with an apology message, the reason for the delay and a revised estimated delivery date.





System Requirements
1.	The system shall be compatible to both IOS and Android version.
2.	The system shall work with 4G, 5G ands Wi-Fi connections and shall be optimised to function on slower network connections where possible.
3.	The system shall restrict all forms of screenshots and screen recording within the app to protect sensitive customer and payment data.
4.	The system shall be capable of handling millions of concurrent users without performance degradation or downtime.
User Requirement
1.	The user shall be able to search for products and view their full specifications, sizes, colours and descriptions on the app.
2.	The user shall be able to track the real-time status of their order from placement through to delivery.
3.	The user shall be able to select their preferred size, colour, and quantity before completing a purchase.
4.	The user shall be able to contact customer support directly through the app in the event of any order issues or complaint.
5.	The user shall be able to initiate a return for incorrectly delivered products within 7 days of delivery.





Assumptions and Constraint
Assumptions
1.	It is assumed that adequate budget and resources will be available throughout the project.
2.	It is assumed that the technical team has the required expertise to implement the system improvement.
3.	It is assumed that users will adapt to and engage with the improved system after launch.
4.	It is assumed that customers have full access to the app via their mobile devices.
5.	It is assumed that the app will be thoroughly tested and validated before launch.
 
Constraints
1.	The project is limited by available funding and budget allocation.
2.	The project must be completed within the agreed timeline.
3.	Technical resources and system compatibility may limit certain implementation options.
4.	The system must comply with applicable data privacy laws across all operating countries, payment security standards, established refund policies and order processing regulations.





Acceptance Criteria
1.	The system will be accepted when it successfully passes the trial testing at least three times without critical errors.
2.	The system will be accepted when all upgraded features are confirmed to be functioning correctly and as intended.
3.	The system will be accepted when all sections of the app have been thoroughly tested and verified by the technical team.
4.	The system will be accepted when all security and payment features have been tested and confirmed to meet required security standards.
5.	The system will be accepted when user acceptance testing has been successfully completed and signed off by the project manager.
